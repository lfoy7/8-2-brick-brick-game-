# 8-2-brick-brick-game-
This project is a brick-destroying game with simple graphics. The game does not have a full objective or scoring system, but it allows the player to create bricks by pressing a button and launch moving balls that interact with them. When the balls collide with the bricks, the bricks are either marked as hit or destroyed depending on their type.
Controls are  as follows: spacebar to make the cirlces and "B" to make the bricks.
I approach designing software by first thinking about what the final product needs to do and what the user should experience. For this project, I started by choosing a desk scene and breaking it down into smaller objects, such as the lamp, book, and coffee mug. Instead of trying to build everything at once, I thought about which basic shapes could represent each object and how those shapes needed to be scaled, rotated, and positioned in the scene.
One new design skill this project helped me build was learning how to plan a 3D scene visually before coding it. I had to think about object placement, proportions, lighting, textures, and how each part of the scene worked together. My design process involved choosing the scene, identifying the main objects, building each object from basic shapes, and then adjusting the details until the scene looked closer to what I wanted. This design approach can help me in future work because breaking a large project into smaller parts makes it easier to manage and improve over time.
I approach developing programs by starting with the basic structure first and then adding more features step by step. While working on my 3D scene, I used development strategies such as testing small changes, adjusting values, debugging errors, and comparing the output to what I expected. Iteration was a big part of the process because the scene did not look right the first time. I had to keep changing scale, position, rotation, and texture settings until the objects looked more natural.
Throughout the milestones, my approach to coding improved because I learned not to rush straight to the final result. Earlier in the course, my attention to detail was not as strong, and small mistakes made objects look wrong or caused the scene not to display correctly. By the end of the project, I became better at checking my code, testing one change at a time, and understanding how each part affected the final scene.
